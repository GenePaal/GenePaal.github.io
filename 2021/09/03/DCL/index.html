最普通的懒汉模式

缺点

  1. 线程不安全。可能会有 > 1 个数量的线程判断到 instance == null成立，造成实例化出 > 1 个对象
  2. 指令重排问题。假设有线程A和线程B，这两个线程“同时”执行 getInstance方法，线程A判断到 instance == null， 然后继续向下执行，实例化对象，但是实例化对象的过程中发生指令重排。正常的实例化顺序：a、为对象开辟内存空间，b、加载非static变量，给非static变量赋默认值(0,null), 赋显式声明的初值 c、执行构造方法 d、令instance指向该对象所在内存的首址 e.完成初始化对象的后续操作 如果指令重排成了 a, d… ， 并且在执行完c后(此时instance不再为null)，cpu的使用权交给了线程B，那么线程B调用 getInstance方法后会返回一个没有初始化的对象，报对象未初始化error。

加单锁

缺点

这样做既保证了线程安全，又可以防止上一个版本的代码中出现的指令重排问题(其实仍然可能会发生指令重排a,c,b，只不过加锁后保证了实例化对象的过程都是发生在一个线程中，不会导致执行a、c后而又让另一个线程调用方法，导致出现异常)。 但是，这种锁的粒度太大，同一时间只能有一个线程执行同步方法拿到这个单例，因此，在高并发环境下，吞吐量严重受限。

还有一个缺陷就是变量没有用volatile修饰，在synchronized方法内 new 对象，修改的是共享的引用，如果在线程A中指令重排了，执行了上述所说的new对象过程中的a、c，还没有来得及执行b，然后这时线程B获取到了CPU使用权，判断最外层那个if，直接返回了一个没有初始化的对象，会报异常。

DCL

解释

1. 在锁外的if判断

减小锁的粒度。这个if判断没有在锁内，目的就是当intance != null 时，可以直接返回对象，而 == null时，进入加锁的方法初始化对象，对上面吞吐量低做了优化。

2. synchronized

保证第二次判断的instance是最新的，保证不会有多个线程“同时”通过第二个if判断检测到 instance为null

3. 在锁内的if判断

有可能线程A通过了最外层的if判断后，还没来的及获取锁，CPU立即交给了线程B，线程B通过最外层的if判断，然后获取锁，最终new出了对象，然后CPU交回给线程A，这时对象已经被线程B实例化了，如果不加第二个if判断，线程A也会执行实例化对象的语句，导致不是单例模式。

4. volatile

禁止指令重排。在synchronized方法内 new 对象，修改的是共享的引用，如果在线程A中指令重排了，执行了上述所说的new对象过程中的a、c，还没有来得及执行b，然后这时线程B获取到了CPU使用权，判断最外层那个if，直接返回了一个没有初始化的对象，会报异常。
