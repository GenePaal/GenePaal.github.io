重写equals()方法

发表于 2022-02-25 |
为什么重写equals()方法也要重写hashcode()？

equals() 和 hashcode() 都是 Object对象的方法。

equals 源码如下：

可以看出原始equals() 方法内部是用 == 作为判断依据，即若两个对象是不是同一个实例/两个对象的引用是否指向同一块内存/两个对象的引用的值是否相等。因此equals返回值为true即可认为两个对象的地址相同。

而原始hashcode() 是根据对象的地址，根据哈希函数计算出来的，若对象地址相同，得到的hashcode必然也相同。

因此在”出厂“时，二者便有了这样一关系：equals返回结果为true的两个对象的hashcode必然相同。同时逆反一下，hashcode不同的两个对象，equals返回结果必然是false。特别说明一下，hash表是可能有冲突的，因此输入不同的地址，hash函数计算出来的数值也是可能相同的，因此，hashcode相同的两个对象不一定是同一个对象(equals方法返回结果不一定是true)。

在map中，判断两个key(obj1, obj2)是否重复的伪码：

( obj1.hashcode() == obj2.hashcode() )

&& ((obj1 == obj2) || obj1.equals(obj2) == true)

hashcode 保证性能。主要是通过那个逆反条件 hashcode不同的两个对象，equals返回结果必然是false(必然不是同一个对象)，再加上 && 符号的特性， 可以很快地判断两个是否必定不是同一个对象。一个map中肯定不是一个对象的要多，因此在 && 前面比较hashcode可以很大幅度地提高比较的速度。

equals 保证可靠。由于hash散列可能有冲突，因此不是同一个对象的hashcode可能也是相同的，因此若hashcode相同，还是要判断后面((obj1 == obj2) || obj1.equals(obj2) == true)

这样，hashcode和equals的同时存在就说的通了，它们互相弥补，以最快地判断两个对象是否相同。两者的存在并不冗余。

那为什么((obj1 == obj2) || obj1.equals(obj2) == true) 后面还要加一个obj1.equals(obj2) == true ？ 这就该引出我们的 重写equals方法了。

在开发中，我们很多时候并不希望通过地址来判断是不是同一个对象，我们希望的结果是，只要两个对象的值相同，我们就认为它们是相同的， 如 我们经常将 String类型的字符串作为key来放入map中，正好String也重写了equals方法， 如果String 不重写equals方法，那么 相同的字符串”abc” 作为key 可能会存两次，在根据”abc” 作为key取时，也可能取到另一个value。 这是我们重写equals方法的初衷。

大前提调用equals方法的对象引用值不为null

  5. obj.equals(null) == false

那为什么重写equals也要重写hashcode方法？

网上对于这块众说纷纭，我琢磨一番给出一个相对本质的答案吧。 如果重写了equals()不重写hashcode() 会影响hashmap中对key是否已存在的判断， 如果你的对象即将作为key值且重写了equals()，那么必须重写hashcode()。 如果你仅仅是希望改变判断对象是否相等时依据，不打算将其作为key值，可以不重写hashcode。

上面说到hashmap判断key是否重复的条件是 ( obj1.hashcode() == obj2.hashcode() )

&& ((obj1 == obj2) || obj1.equals(obj2) == true)， 其实最本质最原始的还是比较地址，加了一个hashcode 是为了提高性能， 加了一个equals是为了在存map时，不将地址作为判断是否为同一个key时，hashmap也能正常工作。

上文说到“出厂”时二者具有这一关系， hashmap判断key是否重复的方法也是根据这一关系设计的， equals比较地址，hashcode就根据地址计算。 当你的equals比较的是值时，hashcode也要根据值来计算。否则会出现，equals相等代表两个对象相等，但是hashcode还是根据地址计算，由于是两个不同的对象，极大概率会不相等，因此极大概率会存两遍。
